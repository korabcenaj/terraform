📋 Case Study: Zero-Downtime Network Migration

🔄 Full Subnet Migration After Router Replacement

Date: May 31, 2026  |  Impact: 3-node cluster, 20+ services, all static IPs

The Challenge

Router replacement shifted the entire cluster subnet from 192.168.1.0/24 → 192.168.0.0/24. Every node, service, static IP, certificate, and DNS record broke simultaneously. Zero access to the API server.

Incident Timeline (50 min to full recovery)

T+0 — All nodes unreachable
dial tcp 192.168.1.13:6443: no route to host. API server down.
T+5 — Diagnosis & IP mapping
Confirmed new subnet via ping/ip route. Mapped new node IPs.
T+10 — Config updates
Updated WSL hosts, kubeconfig, 15+ Terraform variables.
T+15 — Master node recovery
SSH'd in. Fixed /etc/hosts, regenerated API server certs, restarted kubelet.
T+20 — TLS cert mismatch
x509: certificate valid for k8s-master, not k8s-master.local. Fixed kubelet/scheduler/controller-manager conf.
T+30 — Worker nodes recovered
SSH'd into workers, updated hosts + kubelet.conf, restarted kubelet.
T+35 — Kyverno blocking
Deleted validating webhook configurations to unblock recovery.
T+40 — Control plane fixes
Fixed kube-proxy ConfigMap, controller-manager leader election.
T+45 — CNI down
Flannel lost PodCIDR. Manually assigned 10.244.1.0/24, 10.244.2.0/24.
T+50 — ✅ Full recovery
All 3 nodes Ready. 20+ services back. Zero data loss.
